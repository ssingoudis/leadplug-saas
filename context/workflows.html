Workflow 3

Session-Management

Session-Cookie

Nach erfolgreichem Login setzt Supabase einen httpOnly-Cookie mit Access- & Refresh-Token. Jede Server-Anfrage liest diesen Cookie automatisch.

Auth-Check

Jede geschützte Seite (alle /dashboard/*) ruft supabase.auth.getUser() auf dem Server auf. Kein User → Redirect zu /login.

Logout

GET /logout ruft signOut() auf → löscht Session-Cookie → Redirect zu /login.

Route Protection (proxy.ts)

Nicht eingeloggt + geschützte Route Redirect → /login?from=/ursprüngliche-url
Eingeloggt + öffentliche Route (/login, /signup) Redirect → /dashboard

Workflow 5

Datenbank-Interaktionen pro Seite

Route Tabelle / API Operation Client Beschreibung
/dashboard/layout auth.users AUTH Server (RLS) getUser() – prüft Session
/dashboard/layout tenants SELECT Admin (kein RLS) Tenant für auth_user_id laden
/dashboard/layout tenants INSERT Admin (kein RLS) Neuen Tenant anlegen bei erstem Login
/dashboard (Leads) funnels SELECT Server (RLS) slug + total_views aller aktiven Funnels
/dashboard (Leads) submissions SELECT Server (RLS) Letzte 50 Leads + 14-Tage-Chart-Daten
/dashboard (Leads) funnel_questions SELECT Server (RLS) Fragen für Antwort-Anzeige pro Funnel
/dashboard/statistiken submissions SELECT Server (RLS) created_at letzte 12 Monate
/dashboard/statistiken funnels SELECT Server (RLS) total_views (kumulativ) für Conversion
/api/tenant/funnels/[slug]/tracking funnels SELECT + UPDATE Server (RLS) Conversion-IDs (meta_pixel_id, google_ads_conversion) lesen/speichern — Aufgabe 43
/dashboard/account auth.users AUTH Browser (RLS) getUser() → email, user_metadata laden
/dashboard/account auth.users UPDATE Browser (RLS) updateUser() → display_name, phone, password
/auth/confirm auth.users AUTH Server (SSR) verifyOtp(token_hash, type=signup) → E-Mail bestätigen
/auth/callback auth.users AUTH Browser exchangeCodeForSession() → OAuth-Code eintauschen
/api/track-view funnels RPC Server (Service Key) increment_funnel_views(slug) → total_views +1
/api/submit submissions INSERT Server (Service Key) Lead + Antworten speichern
/api/admin/create-funnel tenants · funnels · funnel_questions INSERT Server (Service Key) Neuen Tenant + Funnel + Fragen atomisch anlegen (Admin-only)
RLS (Row Level Security): Supabase filtert automatisch nach tenant_id des eingeloggten Users. Jeder Tenant sieht nur seine eigenen Funnels, Submissions und Statistiken. Admin-Client (Service Key) umgeht RLS – wird nur server-seitig für Tenant-Management verwendet.

Workflow 6

Widget / iFrame – Besucher-Flow (öffentlich)

Kunden-Website <iframe src="/w/[slug]">
→
getTenantConfig() Fragen + Farben laden
→
Funnel ausfüllen Schritte 1–N
→
/api/submit Honeypot-Check
→
submissions INSERT Lead gespeichert
→
2 E-Mails Danke + Lead-Info
Kein Login nötig für Widget-Besucher. /api/submit verwendet den Service Key server-seitig. Honeypot-Feld: bei Auslösung → 200 zurück, aber kein DB-Eintrag. Höhen-Sync via postMessage({type:'funnel-resize', height: X}).
Workflow 7

Admin Funnel-Creator (/admin/new)

/admin/new Client Component
→
Vorschau localStorage → neuer Tab
→
/admin/preview Funnel live prüfen
↓ Speichern
POST /api/admin/create-funnel Service Key
→
INSERT tenants Stammdaten + Billing
→
INSERT funnels Slug + Theme + Texte
→
INSERT funnel_questions Fragen + Optionen + Config
→
/admin/[slug] Redirect
Fragetypen: single_choice, multiple_choice, slider, short_text, long_text. Slider-Config (min/max/step/unit) und Freitext-Config (placeholder, required) werden als JSONB im config-Feld gespeichert. required ist für alle Fragetypen verfügbar. Vorschau schreibt nichts in die DB — Konfiguration wird via localStorage an /admin/preview übergeben.
