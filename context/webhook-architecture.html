1 · Action-Element-Modell — die Kern-Idee

LeadPlug ist „eine Art Typeform-Klon": alle Output-Mechanismen sind dynamisch konfigurierbare Builder-Elemente. Webhooks sind die erste Action-Klasse — E-Mails, Logic-Jumps, Slack-Notifications folgen demselben Pattern.

VORHER (vor Aufgabe 40)
/api/submit → hartkodiert → Mail + (geplanter) Tenant-globaler Webhook.
Tenant konnte nichts steuern. Trigger immer „bei Submit".
JETZT (Aufgabe 40)
Funnel → N Action-Subscriptions im Editor-Tab.
Pro Webhook: wann (on_submit oder after_page:<id>) und welche Events (completed, abandoned, step.advanced).
Tenant entscheidet, wir vorplanen nicht.

2 · Die drei Welten

Der Sender lebt server-side. Cron läuft in Vercel. Tenant-CRM ist außerhalb — Zapier, Make, Pipedream, n8n, oder direkter Endpoint.

① Endkunde-Browser

Live-Widget (iFrame)

Sendet /api/submit beim Funnel-Ende (oder Auto-Finish im Skip-Mode). Sendet /api/track-progress alle 600ms-debounced bei Antwort-Änderungen.

② LeadPlug-Server (Vercel)

Sender + Cron + Editor-API

lib/webhooks.ts baut Payload + HMAC, postet mit Timeout, loggt in delivery_attempts.
after() aus /api/submit triggert fire-and-forget.
Cron alle 5 Min macht Retries + Abbrecher-Trigger.

③ Tenant-CRM

Zapier / Pipedrive / HubSpot / …

Empfängt POST mit JSON-Payload + HMAC-Header. Verifiziert optional die Signatur. Legt Lead im CRM an, triggert Slack-Notify, …

5 · Flow: Completed-Webhook

① Erfolgreiches Submit eines Endkunden — der häufigste Pfad
Browser
LeadPlug-Server
Tenant-CRM
  1. User klickt letzte Antwort. Widget POSTet /api/submit
    ───────────►
  2. Server: Honeypot & Rate-Limit-Check, Validation, getTenantConfig, upsertSubmissionProgress(completed=true) setzt completed_at = NOW(), sendAllEmails, updateEmailStatus
  3. Server: Antwort {success: true} geht sofort raus ◄───────────
    UI kann Erfolgsseite anzeigen, kein Warten auf Tenant-CRM
  4. Server (after-Hook im Hintergrund): triggerOnSubmit(funnelId, 'submission.completed', snapshot, tenantConfig)
  5. Server: SELECT subs WHERE funnel_id=X AND is_active AND trigger_type='on_submit' AND event_types contains event
  6. Server: Parallel pro Match: buildPayload + HMAC-Sig + postWithTimeout(url, body, sig, 10s) ───────────►
  7. Tenant: Zapier empfängt, verifiziert Sig (optional), legt Deal in Pipedrive an, schickt Slack-Notify, antwortet 200 OK ◄───────────
  8. Server: INSERT delivery_attempts {status: 'success', response_status_code: 200, delivered_at: NOW()}

6 · Flow: Abbrecher-Webhook (10 Min Cooldown)

② User füllt Email + Telefon ein, schließt dann Tab — Cron holt nach
Browser
Vercel Cron + Server
Tenant-CRM
  1. t=0: User tippt sich durch — jede 600ms-debounced /api/track-progress UPSERTet submissions-Row mit contact + answers + completed_at=NULL ───────────►
  2. t=14:00: User schließt Tab
  3. Cron läuft alle 5 Min — t=14:15: GET /api/cron/webhook-retry mit Authorization: Bearer $CRON_SECRET
  4. Server: Auth-Check, dann SELECT submissions WHERE completed_at IS NULL AND abandoned_webhook_fired_at IS NULL AND created_at < NOW() - 10min
  5. Server: Pro Match: effectiveContact = derive(answers) ∪ contact
  6. Server: Skip wenn weder email noch telefon → trotzdem Marker setzen, sonst Endlos-Re-Pick
  7. Server: Sonst: triggerOnSubmit(funnelId, 'submission.abandoned', snapshot, tenantConfig) ───────────►
  8. Tenant: Zapier empfängt + verarbeitet wie completed, nur mit "event": "submission.abandoned" + "completed_at": null ◄───────────
  9. Server: UPDATE submissions SET abandoned_webhook_fired_at = NOW() — verhindert Doppel-Trigger im nächsten Cron-Run
Cooldown-Wahl: 10 Min ist ein bewusster Trade-Off zwischen Speed-to-Lead (je schneller der Tenant anruft, desto höher die Recovery-Chance — Studien: <5 Min = 9× höhere Connect-Rate) und falscher Trigger-Vermeidung (User schreibt Email, geht aufs WC, kommt 8 Min später wieder).

7 · Flow: Retry bei Endpoint-Failure

③ Tenant-CRM antwortet 5xx oder Timeout — Cron versucht's nochmal mit Backoff
Cron picked attempt
retryDelivery
Tenant-CRM
  1. Vorlauf: Vorheriger Versuch endete mit status='retrying', attempt_count=1, next_retry_at = NOW + 1min
  2. Cron-Run: SELECT delivery_attempts WHERE status IN ('pending','retrying') AND next_retry_at <= NOW() ORDER BY next_retry_at LIMIT 50
  3. Server: Pro Match → retryDelivery(attemptId)
  4. Server: Re-load attempt + subscription (Join). Skip wenn !is_active → status='failed'
  5. Server: Re-load submission via FK. Wenn gelöscht → status='failed' (audit bleibt)
  6. Server: tenantConfig laden (lazy import um Circular zu vermeiden)
  7. Server: Re-build Payload mit derselben delivery_id (Idempotency auf Tenant-Seite!)
  8. Server: HMAC neu signieren + POST ───────────►
  9. Tenant: 200 OK ◄─────────── oder wieder Fehler
  10. Server: Wenn OK → status='success', delivered_at=NOW(), next_retry_at=NULL. Sonst → attempt_count++ und entweder next_retry_at = NOW + nextRetryDelayMs(N) oder bei N≥6 status='failed'
Backoff-Schedule (Stripe-Klassiker): Versuch 1 ist sofort. Versuch 2 nach 1 Min, 3 nach 5 Min, 4 nach 30 Min, 5 nach 2 h, 6 nach 6 h. Danach failed. Total = ~9 h, deckt typische CRM-Outages ab.

9 · HMAC-Signatur (Stripe-Pattern)

Header X-LeadPlug-Signature: t=<ts>,v1=<hex>. Tenant kann optional die Echtheit verifizieren — Code-Snippets für Node/Python/PHP stehen direkt im Editor-UI.

① LeadPlug-Sender
t = Math.floor(Date.now()/1000)
v1 = HMAC-SHA256(secret,
  "${t}.${bodyJson}")
.toHex()

Header: t=${t},v1=${v1}
→
② Tenant-Server
Header parsen → t + v1
rawBody = req.raw (vor JSON.parse!)
expected = HMAC-SHA256(secret,
  "${t}.${rawBody}")

compare_digest(expected, v1) ?
→ match: trusten
→ mismatch: 401
Optional aber empfohlen: Timestamp gegen now() prüfen (z.B. nur akzeptieren wenn |now - t| < 300 Sek) → Replay-Schutz.
